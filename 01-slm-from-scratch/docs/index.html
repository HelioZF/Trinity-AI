A build-it-yourself field guide

How a small language model is built, one part at a time.

This codex is the growing reference for Trinity-AI, Pillar I — a GPT-2-style model, written from scratch in pure PyTorch, that turns a paper title into a plausible scientific abstract. Each phase is added here after it's understood and built — concepts, the “why”, the code, and live demos.

“Hi” → 72 105 → bytes → merge → tokens → the model

Orientation

The project

The goal is a compact language model that performs conditional text generation: given a paper title, it drafts an abstract. That's expansion, not summarization — a deliberate choice.

Why title → abstract, and not full-paper summarization? Because self-attention cost grows quadratically with sequence length. Summarizing whole papers needs a huge context window, which is infeasible on a single 12 GB RTX 3060. Title → abstract keeps the context small and trainable locally — private, free, and entirely on-device.

Design constraint

The 12 GB GPU is treated as a design parameter, not an afterthought. Model size, vocabulary, batch size, and sequence length are all chosen to fit it — using gradient accumulation and mixed-precision (AMP) training later on.

Method

How this codex works

Two passes over the whole project. Iteration 1 — make it work: learn each phase and get the model training and generating. Iteration 2 — make it clean: refine the code and the writing so any programmer can learn from it. This page is the artifact of that second promise, filled in as we go.

Every phase entry aims to answer four questions, in order: why the part exists, what the theory is, how it's built, and does it work.

Before any modeling: a reproducible environment. A Python virtual environment, PyTorch installed with a CUDA build matched to the GPU, and a check that the GPU is actually visible.

Done when

torch.cuda.is_available() returns True and the environment is pinned in requirements.txt so it rebuilds identically.

Portability note: a CUDA-pinned requirements.txt is machine-specific. On a different GPU (or none), the pinned build won't resolve — that's expected, and revisited in Phase 5.

Phase 1 · complete

Tokenization

Turning text into numbers — and back — with a Byte-Pair Encoding (BPE) tokenizer built from scratch.

A neural network is a math machine: it only operates on numbers. A tokenizer is the bidirectional translator between text and integers. It encodes text into a sequence of integer token IDs (via a vocabulary) so the model can process it, and decodes the model's output IDs back into readable text.

Done when

The round-trip holds — decode(encode(x)) == x for arbitrary text — and the vocabulary is a sensible size that actually compresses the input.

Where the tokenizer lives

The tokenizer is the gate at both ends of the model. Text enters through encode and leaves through decode; everything in between is pure integers. The model core never sees a single character.

The big idea

The model core never sees text — it only ever sees integer token IDs. The tokenizer is the translator standing at both doors.

The tokenizer's responsibilities

  • Train — learn the merges from a corpus, building the vocabulary (once, offline).
  • encode — text → token IDs (the entry gate).
  • decode — token IDs → text (the exit gate).
  • Define the vocabulary size — which sets the model's embedding table and output layer dimensions (both grow linearly with vocab — the Phase-1 tradeoff).

Not its job: embeddings, attention, or learning meaning — those belong to the model. After training, the tokenizer is a fixed, deterministic translator.

Why turn text into numbers?

Symbols mean nothing to a matrix multiply. Every character, word, or sub-word must become an integer the model can look up and learn a vector for. The simplest version — character-level — was used in the makemore exercise (a→1, b→2…). A production tokenizer does the same job on smarter units: subwords.

Unicode, code points, and UTF-8

A code point is the integer Unicode assigns to a character: ord('A') = 65, ord('😀') = 128512. There are ~150,000 of them.

A byte holds one of 256 values, 0–255 (2⁸ = 256 values; counting from 0, the max is 255). A code point like 128512 is far too big for one byte — so code points are encoded into bytes by UTF-8, which uses 1 to 4 bytes per character depending on the code point's magnitude:

CharacterCode pointUTF-8 bytes
A65[65]
é233[195, 169]
😀128512[240, 159, 152, 128]

UTF-8 is chosen over UTF-16/32 for three reasons: it's compact (ASCII stays 1 byte, where the others pad with zero bytes), ASCII-compatible (old text is already valid UTF-8), and it yields a clean byte stream (0–255) — a fixed base alphabet of exactly 256 tokens with no out-of-vocabulary ever. All three encodings cover all of Unicode; the win is efficiency.

The granularity tradeoff

Vocabulary size and sequence length pull in opposite directions. Finer units mean a tiny vocabulary but long sequences; coarser units mean a huge vocabulary but short sequences.

GranularityVocab sizeSequence lengthProblem
Raw bytessmallest (fixed 256)longestsequences far too long
Characterssmalllonglittle meaning per token
Wordslargest (100k+)shortestout-of-vocabulary on unseen words
Subwords (BPE)moderatemoderatethe Goldilocks middle

Subwords keep common words as single tokens and split rare words into pieces — word-like efficiency with no OOV. Two costs scale here, and they matter for a small model on 12 GB: the embedding + output layers grow linearly with vocabulary, while attention grows quadratically with sequence length. Lean toward a moderate vocabulary so the lookup tables don't dominate the parameter budget.

BPE training — learning the merges

Byte-Pair Encoding is greedy and simple. Start from the 256 byte tokens, then repeat: count adjacent pairs, merge the most frequent one into a brand-new token, and record it — until the target vocabulary is reached.

  1. Start: vocab = 256 byte tokens; encode the text to a byte sequence.
  2. Count every adjacent pair.
  3. Pick the winner — the most frequent pair.
  4. Merge: mint a new id (256, 257, …), replace all occurrences, and record the merge in order.
  5. Repeat until vocab_size = 256 + num_merges.

Worked by hand on aaabdaaabac:

RoundWinnerMergeResult
start——aaabdaaabac
1aa (4×)aa → ZZabdZabac
2ab (2×)ab → YZYdZYac
3ZY (2×)ZY → XXdXac

Notice merge 3 (ZY) is built on top of merges 1 and 2. That stacking is why encoding must later replay merges in order — see below.

GPT-2's vocabulary of 50,257 is exactly 256 base bytes + 50,000 merges + 1 special token (<|endoftext|>).

encode / decode — and the one key insight

decode(ids) is a straight lookup: map each id to its bytes, concatenate them, then bytes.decode("utf-8", errors="replace"). The errors="replace" matters — a byte-level model can emit a sequence of bytes that isn't valid UTF-8, and instead of crashing, the invalid part becomes the replacement character �.

encode(text) is the subtle one: convert to bytes, then repeatedly apply the earliest-learned applicable merge — the present pair with the lowest merge index.

The key insight

Training picks merges by frequency. Encoding replays them strictly by order. Why? Because later merges are built on earlier ones (the stacking above), and to reproduce the exact tokenization the model was trained on, the merges must fire in the same order they were created. Train by frequency · encode by order · decode by lookup.

Edge case: encode loops while len(ids) >= 2 — with fewer than two tokens there are no pairs, and taking min over an empty set would raise an error.

The code

The whole tokenizer is six functions of pure standard-library Python — no dependencies. This is the implementation as built in src/tokenizer.py.

def get_stats(ids):
    # count every adjacent pair of ids
    counts = {}
    for p1, p2 in zip(ids, ids[1:]):
        counts[(p1, p2)] = counts.get((p1, p2), 0) + 1
    return counts

def merge(ids, pair, idx):
    # replace every occurrence of `pair` with the new id `idx`
    newids, i = [], 0
    while i < len(ids):
        if (i < len(ids) - 1) and (ids[i], ids[i+1]) == pair:
            newids.append(idx); i += 2
        else:
            newids.append(ids[i]); i += 1
    return newids

def train(text, vocab_size):
    assert vocab_size >= 256
    num_merges = vocab_size - 256
    ids = list(text.encode('utf-8'))
    merges = {}
    vocab = {i: bytes([i]) for i in range(256)}
    for i in range(num_merges):
        stats = get_stats(ids)
        pair = max(stats, key=stats.get)      # winner = most frequent
        idx = 256 + i
        ids = merge(ids, pair, idx)
        merges[pair] = idx                     # recorded in order
        vocab[idx] = vocab[pair[0]] + vocab[pair[1]]  # cumulative bytes
    return merges, vocab

def encode(text, merges):
    ids = list(text.encode('utf-8'))
    while len(ids) >= 2:
        stats = get_stats(ids)
        pair = min(stats, key=lambda p: merges.get(p, float('inf')))  # lowest merge index
        if pair not in merges:
            break
        ids = merge(ids, pair, merges[pair])
    return ids

def decode(ids, vocab):
    tokens = b"".join(vocab[idx] for idx in ids)
    return tokens.decode('utf-8', errors='replace')

Why LLMs act weird — it's often the tokenizer

Many famous failures aren't the model being unintelligent; they're artifacts of tokenization:

SymptomTokenizer cause
Bad spelling / reversing wordsletters are hidden inside multi-char tokens, so the model can't see them
Weak arithmeticnumbers split inconsistently (677 vs 6+77)
Worse in other languagesmore tokens per word → less efficient, less effective training
“SolidGoldMagikarp” glitchesrare tokens that got an id but were barely trained → unstable embeddings

Real GPT tokenizers also add a regex pre-split (so merges never cross letter/number/ punctuation boundaries) and special tokens like <|endoftext|>, inserted outside the merge process. Libraries: tiktoken (bytes-first, inference-only) vs SentencePiece (code-points-first, can train).

Reference

Glossary

Token
An integer id standing for a chunk of text (a byte, character, or sub-word), looked up in the vocabulary.
Code point
The integer Unicode assigns to a character (e.g. A = 65). Distinct from a byte.
UTF-8
A variable-length encoding (1–4 bytes/char) that turns code points into a byte stream.
BPE
Byte-Pair Encoding — repeatedly merges the most frequent adjacent pair of tokens into a new token.
Merge
A learned rule (a, b) → new_id. Merges are recorded in training order.
Vocabulary
The map from token id to its bytes; its size is a hyperparameter (256 + num_merges).
Round-trip
The correctness property decode(encode(x)) == x.
OOV
Out-of-vocabulary — a unit the tokenizer can't represent. Byte-level BPE never has this.
